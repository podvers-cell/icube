# اقتراحات لتحسين موقع ICUBE Media Studio

بناءً على تقارير الحالة والتدقيق والحالة الحالية للمشروع.

---

## ١. تحسينات الأداء (Performance)

| الاقتراح | الفائدة | الجهد |
|----------|---------|--------|
| **تحميل الداشبورد عند الحاجة فقط** | تقليل حجم الـ bundle الأولي للزائر العادي؛ تحميل Firebase وواجهة الداشبورد فقط عند دخول `/dashboard`. | متوسط – استخدام `dynamic(..., { ssr: false })` أو route-based code splitting. |
| **تقسيم جلب البيانات في SiteDataContext** | بدل جلب كل المجموعات دفعة واحدة، جلب الأساسيات أولاً (الإعدادات، الهيرو) ثم الباقي عند الحاجة أو عند الوصول لأقسام معينة. | متوسط. |
| **استكمال استخدام next/image** | تم تحسين صور الاستوديو؛ يمكن توسيعه لبقية الصور (الهيرو، البورتفوليو، الشهادات) مع `sizes` و `priority` حيث يلزم. | منخفض. |
| **تحميل الفيديو في الهيرو بشكل كسول** | إذا كان الفيديو تحت الطيّ، تأخير تحميله حتى يصبح قريباً من الظهور أو بعد تفاعل المستخدم. | منخفض. |
| **تفعيل ضغط وcache من الاستضافة** | التأكد من تفعيل Gzip/Brotli و Cache-Control للـ static assets في Vercel أو الخادم. | منخفض (غالباً مفعّل افتراضياً). |

## ٢. تحسينات SEO والمشاركة

| الاقتراح | الفائدة | الجهد |
|----------|---------|--------|
| **روابط Canonical لجميع الصفحات العامة** | تجنب محتوى مكرر مع وجود query params أو نسخ من الموقع. | منخفض – إضافة `alternates.canonical` في metadata لكل صفحة. |
| **Open Graph و Twitter Cards لجميع الصفحات** | مشاركة أفضل على وسائل التواصل (صورة، عنوان، وصف) لكل من الصفحة الرئيسية، الاستوديو، البورتفوليو، الحجز، إلخ. | منخفض – توحيد metadata مع `openGraph` و `twitter`. |
| **تحسين عناوين ووصف الصفحات** | عناوين فريدة ووصف واضح لكل صفحة لتحسين النقر من محركات البحث. | منخفض. |
| **بيانات منظمة (JSON-LD)** | إضافة `Organization` و `LocalBusiness` وربما `Product` للاستوديوهات لتحسين ظهور الموقع في البحث. | متوسط. |
| **مراجعة Sitemap و robots** | التأكد أن `sitemap.ts` و `robots.ts` يشملان كل المسارات العامة وأن base URL من المتغيرات (APP_URL). | منخفض (تم جزئياً). |

## ٣. تجربة المستخدم (UX)

| الاقتراح | الفائدة | الجهد |
|----------|---------|--------|
| **سحب (Swipe) للكراسي على الموبايل** | تحريك البطاقات بإصبع المستخدم في أقسام Services، Portfolio، Testimonials، Studio يقلل الاعتماد على الأزرار فقط. | متوسط – استخدام مكتبة gesture أو touch events. |
| **مؤشر تقدم أو CTA ثابت في نماذج الحجز** | في صفحات Date/Time و Add-ons و Checkout، شريط تقدم أو زر "متابعة" ثابت يوضح أين المستخدم ويسهّل الإكمال. | منخفض. |
| **توحيد رسائل النجاح (Toast)** | بدل نصوص داخل الصفحة أو alert، استخدام toast موحّد لـ "تم إرسال الرسالة" و "تم الحجز" ليكون أوضح وأقل إزعاجاً. | منخفض – إضافة مكون Toast واستدعاؤه من النماذج. |
| **حالة تحميل أوضح للداشبورد** | بدل "Loading…" فقط، عرض هيكل الصفحة (skeleton) أو اللوجو مع رسالة تحميل. | منخفض. |
| **تحسين تجربة المودالات على الموبايل** | التأكد أن إغلاق المودال سهل (زر واضح، ضغط خارج المحتوى)، وأن التركيز لا يعلق خلف المودال (focus trap + إرجاع التركيز عند الإغلاق). | متوسط. |

## ٤. إمكانية الوصول (Accessibility)

| الاقتراح | الفائدة | الجهد |
|----------|---------|--------|
| **احترام prefers-reduced-motion** | تقليل أو إيقاف الحركات للمستخدمين الذين يفضلون تقليل الحركة (دوار، إزعاج بصري). | منخفض – قواعد CSS وربما تعطيل بعض حركات motion. |
| **تسميات واضحة لأزرار الكراسي** | `aria-label` مثل "الشهادة السابقة" / "الشهادة التالية" بدل "السابق" / "التالي" فقط. | منخفض. |
| **Focus trap في المودالات** | عند فتح Contact أو Studio أو الفيديو، حبس التركيز داخل المودال وإرجاعه للعنصر الذي فتح المودال عند الإغلاق. | متوسط. |
| **التحقق من تباين الألوان** | التأكد أن الذهبي والنص على الخلفيات الداكنة/الفاتحة يحققان WCAG AA. | منخفض. |
| **aria-live لحالات التحميل** | إعلان تغيير "جاري التحميل" / "تم التحميل" لقارئات الشاشة. | منخفض. |

## ٥. الأمان والتحقق

| الاقتراح | الفائدة | الجهد |
|----------|---------|--------|
| **تحديد معدل الطلبات (Rate limiting)** | حماية نماذج التواصل والحجز وواجهة الرفع من السبام والاستخدام المفرط (مثلاً عبر Vercel middleware أو دالة على السيرفر). | متوسط. |
| **التحقق من صحة المدخلات على السيرفر** | استخدام Zod (أو غيره) في API routes أو Server Actions للتحقق من هيئة الطلبات قبل الكتابة إلى Firestore. | متوسط – توحيد المخططات المستخدمة في الواجهة. |
| **قواعد أمان Firestore** | التأكد أن قراءة/كتابة مجموعات الداشبورد مسموحة فقط للمستخدمين المصرح لهم (مثلاً أدمن). | عالي الأهمية – تنفيذ يدوي في Firebase Console. |
| **رفع الملفات بأمان أقوى (اختياري)** | الاعتماد على Firebase ID token + firebase-admin للتحقق من أن الطالب مسجل دخول كأدمن بدل مفتاح مشترك فقط. | متوسط. |

## ٦. المحتوى والقانوني

| الاقتراح | الفائدة | الجهد |
|----------|---------|--------|
| **إكمال صفحة الخصوصية** | استبدال النص المؤقت في `/privacy` بنص نهائي يصف جمع البيانات والكوكيز والطرف الثالث. | يعتمد على المحتوى القانوني. |
| **إكمال صفحة الشروط** | استبدال النص المؤقت في `/terms` بشروط استخدام الخدمة والحجز. | يعتمد على المحتوى القانوني. |
| **ربط النشرة الإخبارية** | ربط نموذج الاشتراك في الفوتر بخدمة (مثل Mailchimp أو Firestore) أو الإبقاء على "قريباً" مع توضيح للمستخدم. | متوسط إن تم الربط. |

## ٧. الصيانة وجودة الكود

| الاقتراح | الفائدة | الجهد |
|----------|---------|--------|
| **تقسيم المكونات الكبيرة** | تقسيم `Studio.tsx` و `VideoPlayerModal.tsx` إلى مكونات أصغر (بطاقة استوديو، كاروسيل موبايل، محتوى المودال) لتسهيل الاختبار والصيانة. | متوسط. |
| **طبقة نماذج/تحقق مشتركة** | استخدام React Hook Form (أو غيره) مع نفس مخططات Zod للتحقق في الواجهة وربما في API لتفادي تكرار المنطق. | متوسط. |
| **اختبارات E2E للمسارات الحرجة** | تغطية الحجز، التواصل، تسجيل الدخول → الداشبورد بأدوات مثل Playwright أو Cypress لضمان عدم كسر التدفقات بعد التعديلات. | متوسط إلى عالي. |
| **توثيق متغيرات البيئة والنشر** | توثيق واضح لـ APP_URL و UPLOAD_API_KEY و Firebase و Cloudinary في README أو دليل النشر. | منخفض. |

## ٨. خارطة طريق مقترحة (حسب الأولوية)

### المرحلة الأولى (قصيرة المدى)
- إكمال محتوى الخصوصية والشروط.
- إضافة Rate limiting للنماذج وواجهة الرفع.
- توحيد Open Graph و Canonical لجميع الصفحات العامة.
- إضافة Toast موحّد وربما مؤشر تقدم في الحجز.

### المرحلة الثانية (متوسطة المدى)
- تحميل الداشبورد عند الحاجة (code splitting).
- تحسين إمكانية الوصول: prefers-reduced-motion، aria-labels، focus trap.
- إضافة Swipe للكراسي على الموبايل.
- تقوية التحقق (Zod على السيرفر) وقواعد Firestore.

### المرحلة الثالثة (أطول مدى)
- اختبارات E2E للحجز والتواصل والداشبورد.
- تقسيم المكونات الكبيرة وطبقة نماذج مشتركة.
- بيانات منظمة (JSON-LD) وربط النشرة الإخبارية إن لزم.
